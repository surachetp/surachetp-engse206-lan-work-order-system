# A01 — Problem Framing Canvas

> กิจกรรมนี้ช่วยให้ทีมเริ่มจาก “ปัญหา” ไม่ใช่เริ่มจาก “อยากทำระบบอะไร”

## วัตถุประสงค์

- เห็นภาพสถานการณ์ปัจจุบันของงาน
- อธิบาย pain point ที่แท้จริง
- แยก business goal ออกจาก user goal
- เตรียมข้อมูลสำหรับกรอก `P01_problem_brief.md`

## เวลาแนะนำ

45–60 นาที

## วิธีทำ (ทำเป็นทีม)

### Step 1: ระบุสถานการณ์ปัจจุบัน
ตอบคำถามต่อไปนี้ให้สั้น กระชับ และใช้ภาษาที่เข้าใจง่าย

1. ตอนนี้ใครกำลังทำงานอะไรอยู่?
2. งานนั้นทำที่ไหน/อย่างไร?
3. จุดไหนที่สะดุด ล่าช้า ผิดพลาด หรือสื่อสารคลาดเคลื่อน?
4. ผลกระทบเกิดกับใครบ้าง?

### Step 2: เขียน Problem Statement
ใช้รูปแบบนี้

> “ปัจจุบัน [ผู้ใช้/หน่วยงาน] ต้อง [งานที่ทำ] โดยใช้ [วิธีเดิม] ทำให้เกิด [ปัญหา] ส่งผลให้ [ผลกระทบ]”

### Step 3: แยก Goal ออกเป็น 2 ระดับ

| ประเภท goal | คำอธิบาย | ตัวอย่าง |
|---|---|---|
| Business Goal | เป้าหมายเชิงงาน/หน่วยงาน | ลดความซ้ำซ้อนของการจองทรัพยากร |
| User Goal | เป้าหมายของผู้ใช้ปลายทาง | จองห้องได้เร็วและรู้ผลอนุมัติชัดเจน |

### Step 4: ระบุ Pain Points

อย่างน้อย 3 ข้อ โดยเน้นพฤติกรรมหรือสถานการณ์จริง เช่น

- ต้องถามหลายคนกว่าจะรู้ว่าห้องว่างไหม
- เอกสารอนุมัติกระจัดกระจายและติดตามยาก
- ผู้ใช้ไม่รู้เงื่อนไขการจองล่วงหน้า

### Step 5: ระบุ Success Indicators

ตอบว่า “ถ้าปัญหานี้ดีขึ้น เราจะเห็นอะไรเปลี่ยนไป?”

ตัวอย่าง

- ผู้ใช้เห็นสถานะทรัพยากรและส่งคำขอจองได้ในที่เดียว
- เจ้าหน้าที่ติดตามคำขอและประวัติการจองได้ง่ายขึ้น
- ความเข้าใจผิดเรื่องเวลาว่างและการอนุมัติลดลง
